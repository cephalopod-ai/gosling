## 3. Boundary-contract cards (P0/P1)

### B1 — ACP `_gosling/*` custom-method dispatch  (P1)
- Input contract: typed request struct per method (e.g. `GoslingToolCallRequest`),
  method name derived from `#[request(method=…)]`.
- Output contract: typed response struct (`…Response`) or `EmptyResponse`.
- Error contract: `agent_client_protocol::Error` — `invalid_params` (deser
  failure, `acp-macros/src/lib.rs:94-96`), `internal_error` (serialize failure),
  `method_not_found` (unknown method, `lib.rs:191`). Consistent envelope.
- Invariants: unknown method → `method_not_found` (fail-closed); malformed params
  → `invalid_params` with error string in `.data`.
- Trace propagation: failures logged with `method` (`custom_dispatch.rs:14`).
- Pinning: schema round-trip enforced by B2 CI check. **Held.**

### B3 — gosling-as-ACP-client parsing untrusted subprocess output  (P1)
- Input contract: `SessionNotification { update: SessionUpdate }`, content as
  typed `ContentBlock` enum variants (`acp/provider.rs:789-939`).
- Unknown update variants → `_ => {}` (ignored, fail-safe), tool updates only
  emitted on terminal status. No raw `serde_json::Value` field access on the
  untrusted path. **Held** — typed all the way; see Non-Findings.

### B5 — ACP app tool-call entrypoint  (P0)  → **CTR-GSL-001**
- Input contract: `GoslingToolCallRequest { session_id, name, arguments:Value }`.
- Invariant that SHOULD hold: a tool the operator marked `NeverAllow`, or one a
  `PreToolUse` policy hook denies, must not execute.
- Authn/z expectation: **assumed already done — but no layer performs it on this
  path.** Only `is_tool_visible_to_app` (a UI-visibility metadata gate) runs.
- Pinning test: **NONE found.** → IAPI-007 / IAPI-015 (see finding).

### B7 — PermissionDecision ↔ ACP outcome  (P1)
- `PermissionDecision::from(&RequestPermissionOutcome)` parses `option_id` via
  `from_str(...).unwrap_or(Self::Cancel)` (`acp/common.rs:60`) — unknown id →
  Cancel (fail-closed). `map_permission_response` matches by
  `PermissionOptionKind` with graceful fallback. Unit-tested incl.
  `unknown_maps_to_cancel` (`common.rs:213`). **Held.**

## 4. Findings

### CTR-GSL-001: ACP `tools/call` entrypoint executes tools bypassing the permission-inspector and PreToolUse policy hooks

Severity: High (if the ACP client boundary is treated as semi-trusted / reachable) — Medium (single-local-operator posture)
Confidence: Confirmed (bypass mechanism) / Likely (runtime execution of a denied tool — not test-reproduced)
Evidence basis: source-evidenced
Domain: Architecture (IAPI-007 bypassed service layer / IAPI-015 policy not propagated across entrypoints)

Evidence:
- `crates/gosling/src/acp/server/tools.rs:60-104` — `on_call_tool` resolves the
  tool, checks only `is_tool_visible_to_app(tool)` (`tools.rs:72`), then calls
  the **raw** executor directly:
  `agent.extension_manager.dispatch_tool_call(&ctx, tool_call, CancellationToken::new())`
  (`tools.rs:97`). `ctx = ToolCallContext::new(session_id, None, None)`.
- `crates/gosling/src/agents/reply_parts.rs:576-590` — `is_tool_visible_to_app`
  returns `true` whenever `_meta.ui.visibility` is absent (the default); it is a
  display filter, **not** an authorization check.
- The agent loop's enforced path instead runs the permission inspector first:
  `agents/agent.rs:2197` `process_inspection_results_with_permission_inspector`
  → builds `PermissionCheckResult` → `handle_approved_and_denied_tools`
  (`agent.rs:730`) dispatches **only** `permission_check_result.approved`
  through `Agent::dispatch_tool_call` (`agent.rs:743`).
- `permission/permission_inspector.rs:156-159` — `PermissionLevel::NeverAllow`
  → `InspectionAction::Deny`. This runs in the loop path, not on B5.
- `Agent::dispatch_tool_call` (`agent.rs:960-1004`) additionally enforces
  `PreToolUse` hooks: `HookDecision::Deny` → hard error
  ("policy denial, not a transient failure"). The app path never calls this
  wrapper — it calls `extension_manager.dispatch_tool_call` one layer below.
- Caller census: `on_get_tools` (`tools.rs:23-38`) *computes and surfaces* the
  per-tool `NeverAllow`/`AskBefore`/`AlwaysAllow` level to the app, but
  `on_call_tool` never consults `permission_manager` before executing.

Observed behavior:
- Any tool that is visible to the app (default: all tools) can be executed via
  `_gosling/unstable/tools/call` without the permission-inspector verdict
  (including `NeverAllow`) and without `PreToolUse` deny-hook evaluation.

Expected boundary:
- Both entrypoints that execute a tool must run the same policy gate. The
  permission level the app is shown in `on_get_tools` is the exact policy that
  `on_call_tool` fails to enforce.

Failure mechanism:
- Two `dispatch_tool_call` methods exist: `Agent::dispatch_tool_call`
  (loop, guarded) and `ExtensionManager::dispatch_tool_call`
  (`agents/extension_manager.rs:1767`, raw). The app handler binds to the raw
  one, so the guard is structurally skippable with no type error.

Break-it angle:
- Operator sets a tool to `NeverAllow` (e.g. a shell/edit tool). The tool list
  correctly reports `NeverAllow`, yet an ACP client issuing `tools/call` for it
  reaches the extension and runs it. Same for a `PreToolUse` hook that denies a
  destructive command. (Traced statically; execution outcome Likely, not run.)

Impact:
- The stated safety controls (permission store `NeverAllow`, PreToolUse policy
  hooks) do not cover a second, live tool-execution entrypoint. Destructive or
  exfiltration-capable tools can run outside the gate.

Operational impact:
- Blast radius: Local → Workflow (the operator's workstation; tool blast radius).
- Side-effect class: process / file / network / external API (tool-dependent).
- Reversibility: irreversible (depends on tool).
- Operator visibility: silent (no permission prompt; not obviously logged as a
  bypass).
- Rerun safety: unsafe (re-invoking re-executes).

Adjacent failure modes:
- The app path also skips `Agent::dispatch_tool_call`'s subdirectory-hint
  tracking and frontend-tool guard (`agent.rs:1006-1034`) — lower severity but
  same root cause (binding to the raw executor).
- Cross-lens: the *correctness* of the permission rules and the threat model of
  who may open the ACP socket belong to `audit-security`; this finding is the
  seam-parity defect (the decision is not propagated), not the rule itself.

Recommended mitigation:
- Remediation patterns: single-choke-point execution; cross-entrypoint policy
  parity.
- Minimal repair: in `on_call_tool`, before dispatch, consult
  `permission_manager.get_user_permission(&tool.name)` (and smart-approve/approve
  mode as `on_get_tools` already does) and reject `NeverAllow`; route through a
  shared helper that also emits `PreToolUse` hooks — or call
  `Agent::dispatch_tool_call` rather than the extension-manager method.
- Local guardrail: make `ExtensionManager::dispatch_tool_call` callers explicit
  (doc/`#[doc(hidden)]` or a `PolicyGated`/`Unchecked` wrapper type) so a raw
  call is a visible decision.
- Behavior test: with a tool set to `NeverAllow`, `_gosling/unstable/tools/call`
  returns a policy-denial error and the extension is never invoked (assert on a
  spy that the extension received zero calls); with a `PreToolUse` Deny hook, the
  same.

Implementation assessment:
- Complexity: workflow_protocol. Cost: S. Cost drivers: modules, tests.
- Nominal implementation agent: codex.
- Rationale: one handler plus a shared gate helper and two parity tests; the
  security decision of whether `AskBefore` should hard-deny vs elicit on this
  path needs a human-owner call.

Validation:
- Cross-entrypoint parity test: same tool + same permission state via loop path
  and via `tools/call` yields identical policy outcome (both deny for
  `NeverAllow`/hook-deny).

Non-goals:
- Do not redesign the permission model or the MCP-Apps visibility semantics in
  this slice.

## 5. IAPI-001..016 walk — findings & explicit non-findings

- **IAPI-001 Untyped boundary** — *Mostly Held.* The `_gosling/*` surface is
  fully typed (`custom_requests.rs`); the proc-macro deserializes params into the
  concrete request type at the seam (`acp-macros/src/lib.rs:94`). Remaining
  `serde_json::Value` fields are intentional opaque pass-throughs of MCP/arbitrary
  content: `GoslingToolCallResponse.content` (`custom_requests.rs:107`),
  `ConfigRead*Response.value` (`:461,:491`), `DiagnosticsGetResponse.report`
  (`:203`), `ReadResourceResponse.result` (`:88`). These wrap foreign content
  whose shape gosling does not own — not a type-erosion defect. Non-finding.
- **IAPI-002/003 DTO leakage** — *Not Confirmed.* The SDK exposes hand-written
  `*Dto` types (`ProviderConfigFieldValueDto`, `ProviderSecretDto`, …) rather than
  persistence/provider SDK objects; providers return the internal `ModelInfo`/
  `ProviderMetadata` types (`base.rs:22,221`), not vendor SDK structs. No provider
  DTO traced crossing a repo-owned boundary in the sampled surface.
- **IAPI-004 Duplicate schema definitions** — *Held with a caveat.* The Rust↔TS
  duplication is a **generated** contract with a forcing mechanism (B2). Separate
  caveat: gosling-server exposes an OpenAPI surface overlapping some concepts
  (providers/sessions) — a *second transport*, not a drift-uncontrolled duplicate;
  desktop is steered off it (`AGENTS.md`). Deeper assessment routes to
  `audit-architecture-seam`. Not Reviewed here.
- **IAPI-005/006 Error shape** — *Held.* ACP dispatch uses one envelope
  (`agent_client_protocol::Error`, kinds `invalid_params`/`internal_error`/
  `method_not_found`) consistently (`acp-macros/src/lib.rs`, `dispatch.rs:113,147`,
  `tools.rs:69,81`). Failure classes stay distinguishable (invalid_params vs
  internal_error). No thrown-vs-result mixing at these seams.
- **IAPI-007 Bypassed layer** — **CTR-GSL-001** (Confirmed).
- **IAPI-008 Hidden singleton coupling** — *Not assessed deeply.* `Config::global()`
  is used at seams (`dispatch.rs:121`); ambient-global coupling review deferred to
  `audit-architecture-seam`. Recorded as Not Reviewed.
- **IAPI-009 Rules duplicated across layers** — Partially: `on_get_tools`
  re-derives permission display while `on_call_tool` omits enforcement — captured
  under CTR-GSL-001 rather than as a separate ID.
- **IAPI-010 Versionless evolution** — *Held (static).* Convention is the
  `_gosling/unstable/*` method prefix + `_unstable` generated-type suffix
  (`generate_acp_schema.rs:421-431`). B2 CI diff forces all callers' generated
  types to move together. `git log` correlation not runnable (no git).
- **IAPI-011 Nullable/optional drift** — *Not Confirmed* in sampled types;
  Option fields use `skip_serializing_if` and `#[serde(default)]` consistently. No
  unguarded deref traced. Full census not done (Validation Limit).
- **IAPI-012 Lifecycle/construction** — *Held* for the sampled contract types
  (plain serde DTOs; no factory-enforced invariant bypassed). Tool-call terminal
  status handled fail-safe in `acp/provider.rs:844-936`.
- **IAPI-013 Contract test absent / mock-on-mock** — Partial: B2 is CI-enforced
  (strong). B5 (`tools/call`) has **no** pinning/parity test (search:
  `rg "tools/call|on_call_tool" crates/gosling/tests` — not found in inventory;
  see Validation Limits). Feeds CTR-GSL-001.
- **IAPI-014 Trace propagation** — Not the focus of this lens for gosling (no
  request-id correlation model surfaced at these seams beyond `tracing` spans);
  routed to `audit-operator-signal`. `dispatch_custom_request` logs `method` on
  error only (`custom_dispatch.rs:14`).
- **IAPI-015 Policy not propagated across seams** — **CTR-GSL-001** (Confirmed).
- **IAPI-016 Validation drift between entrypoints** — Adjacent to CTR-GSL-001;
  `on_call_tool` validates `arguments` is object/null (`tools.rs:77-84`) but does
  not run the loop's permission validation. Same root cause.

## 6. Key non-findings (checked and held)

- **Codegen contract is CI-enforced (B2).** `Justfile:158` `check-acp-schema`
  runs `generate-acp-types` then `git diff --exit-code crates/gosling/acp-schema.json
  crates/gosling/acp-meta.json ui/sdk/src/generated/`; wired into CI job
  `schema-check` (`.github/workflows/ci.yml:167,195`). Rust type changes that are
  not regenerated fail CI — a real forcing function against Rust↔TS drift
  (IAPI-004/010). The TS client additionally Zod-validates responses
  (`generate-schema.ts:232` `z<Type>.parse(raw)`), fail-closed at runtime.
- **Macro dispatch is fail-closed.** Unknown method → `method_not_found`
  (`acp-macros/src/lib.rs:191`); param deser failure → `invalid_params`.
- **Unknown-enum handling fails closed.** `PermissionDecision::from` unknown
  `option_id` → `Cancel` (`acp/common.rs:60`, tested `common.rs:213`); unknown
  `set_config_option` id → `invalid_params` (`dispatch.rs:145-148`).
- **Untrusted subprocess parsing stays typed.** `acp/provider.rs:789-939` matches
  typed `SessionUpdate`/`ContentBlock` variants; unknown variants ignored.
- **Provider trait is a coherent internal contract.** `trait Provider`
  (`base.rs:395-613`) with typed `ProviderError`, default methods for capability
  negotiation (`manages_own_context`, `permission_routing`,
  `skip_canonical_filtering`); errors are a single typed enum, no `Box<dyn Error>`
  leakage at the trait boundary.
