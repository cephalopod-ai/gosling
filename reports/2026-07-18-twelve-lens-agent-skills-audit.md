# Twelve-lens agent-skills audit — 2026-07-18

Repository: `cephalopod-ai/gosling`  
Revision audited: `05ae05c7b10ead7c66e5f8c52fb217a1d10ba2e7` (`main`, equal to `origin/main`)  
Authority: read-only audit; only this report and its machine-readable companion were added  
Evidence basis: static source/configuration/documentation review; no runtime, build, test, or crash drill was performed

## Executive verdict

The current tree has materially stronger recovery behavior than the earlier July audits: the
session database is transactional, workspace metadata and credential-profile metadata have
separate persistence boundaries, streamed chat messages are checkpointed, artifact routing is
revision-guarded, and schema v23 now gives tool calls a durable operation ledger with explicit
`in_doubt` recovery instead of automatic redispatch.

Ten material defects remain after de-duplication:

| Severity | Count | Finding IDs |
|---|---:|---|
| Critical | 0 | — |
| High | 4 | AUD-GOS-001 through AUD-GOS-004 |
| Medium | 4 | AUD-GOS-005 through AUD-GOS-008 |
| Low | 2 | AUD-GOS-009 through AUD-GOS-010 |

The highest-priority defect is AUD-GOS-001. The Desktop feature named **Local Secret Profiles**
does not use Gosling's secure secret store. It keeps raw values in `settings.json`, returns them
to the renderer, holds them in long-lived React state, embeds them as `KEY=value` prompt text,
persists that message in `sessions.db`, records it into tracing fields, passes it to prompt hooks,
and sends it to the configured model provider. This is a direct credential-confidentiality
failure, not merely missing hardening.

Two other High findings are alternate-path policy bypasses: a renderer can promote any existing
directory into a filesystem authority root (AUD-GOS-002), and Workspace folder `read` versus
`read_write` is prompt guidance rather than an enforced session/tool policy (AUD-GOS-003).
AUD-GOS-004 is the already-routed ORCH-002 design gap, reconfirmed in current source: delegated
roles inherit the parent's complete extension surface by default and always run in Auto mode.

## Findings

### AUD-GOS-001 — Local Secret Profiles are plaintext prompt material, not secure references

Severity: **High**  
Confidence: **Confirmed**  
Evidence basis: source-evidenced  
Domain: Security  
Status: new in this audit

Evidence:

- `ui/desktop/src/utils/settings.ts:8-24,49-60` defines every managed secret as a `value: string`
  inside the ordinary `Settings` object. The same settings object contains the external backend
  bearer `secret` at lines 1-5.
- `ui/desktop/src/main.ts:2021-2051` returns any allowlisted setting to the renderer, accepts the
  replacement value as `unknown`, and writes the complete object to `settings.json`.
- `ui/desktop/src/components/settings/auth/ManagedSecretProfilesSection.tsx:260-274,401,600-619`
  loads complete profiles into React state, persists the same profiles, and binds stored values
  back into the input.
- `ui/desktop/src/components/ChatInput.tsx:101-131` serializes entries as
  ```${entry.key}=${entry.value}```; lines 362-403 keep all profiles in chat state and lines
  453-458 attach the plaintext block as assistant context.
- `ui/desktop/src/types/message.ts:255-262` converts that block into assistant-audience message
  content. `Message::as_concat_text` at
  `crates/gosling-providers/src/conversation/message.rs:943-948` concatenates every text block
  without an audience filter. `crates/gosling/src/agents/agent.rs:1722-1724` records that text into
  tracing fields, lines 1773-1784 pass it to user-prompt hooks, and lines 1889-1892 persist the
  complete user message before provider execution. `Message::agent_visible_content` at
  `crates/gosling-providers/src/conversation/message.rs:794-804` deliberately retains the block
  for the model.

Observed: a saved password, API key, admin token, or private configuration value has at least six
plaintext copies/consumers: Desktop settings, renderer state, the user-message record, tracing,
prompt hooks, and the provider request.

Expected boundary: renderer-facing profiles contain metadata and configured status only. Raw
values remain in Gosling's existing secure secret-storage abstraction, are never persisted in
messages/settings or emitted to logging/hooks, and are resolved at the narrow backend operation
that needs them. A model prompt is not a credential transport.

Failure mechanism: the UI models credentials as ordinary application data and implements
"sharing" by prompt concatenation. Audience annotations hide the block from the user-facing
rendering but do not create a confidentiality boundary; they ensure the assistant/provider sees
it.

Break-it angle: save a sentinel admin credential, select the profile for a chat, submit any
message, then search `settings.json`, `sessions.db`, tracing output, hook payloads, and the provider
request for the sentinel.

Impact: compromise of third-party/API/VPS credentials, leakage through model providers or hooks,
and durable secret residue in files that are not Gosling's protected secret store. Blast radius
is cross-system and effects may be irreversible after a credential is used.

Recommended remediation:

- Remove `value` from renderer/persisted profile contracts and migrate every saved value to the
  existing secure secret store using profile-specific identifiers.
- Make the renderer receive metadata only (`configured`, field names, timestamps). Do not
  pre-populate stored values.
- Resolve a selected profile in the backend for a specific provider/tool/service operation. Do
  not place raw values in ACP messages, prompts, URLs, logs, traces, or hook contexts.
- Migration must be transactional enough to avoid deleting the plaintext source before secure
  storage succeeds; after success, atomically rewrite settings without the values.

Behavior guardrail: a sentinel test must assert absence from settings, renderer responses,
message serialization/session DB, exported sessions, traces, hook payloads, and provider prompt
text.

Implementation assessment: **L**, `workflow_protocol` + secure migration; nominal owner
`multi-agent` with human security/product approval. Non-goal: redesigning provider keyring
internals.

### AUD-GOS-002 — Renderer code can self-authorize an arbitrary filesystem root

Severity: **High**  
Confidence: **Confirmed**  
Evidence basis: source-evidenced  
Domain: Security  
Status: new in this audit

Evidence:

- `ui/desktop/src/main.ts:241-258` treats `loadRecentDirs()` and `archiveFolder` as approved
  filesystem roots.
- `ui/desktop/src/main.ts:1980-1984` accepts any renderer-supplied string for
  `add-recent-dir`; `ui/desktop/src/utils/recentDirs.ts:53-82` requires only that it already
  exists, is a directory, and is not itself a symlink.
- `ui/desktop/src/preload.ts:217-229,368-370` exposes root mutation plus read, write, delete,
  mkdir, and list operations to renderer JavaScript.
- `ui/desktop/src/main.ts:2410-2424,2479-2515` authorizes those operations using the recent-dir
  roots. `ui/desktop/src/utils/rendererFileAccess.ts:59-72` correctly canonicalizes and checks
  containment, but it trusts the supplied root set.

Observed: `addRecentDir('/')` (or an existing sensitive user directory) makes that directory an
approved root without a main-process chooser result, backend workspace grant, or per-window
capability. The containment function then correctly—but dangerously—authorizes every descendant.

Expected boundary: renderer input may request access but cannot mint authority. The main process
must derive grants from a real directory chooser or an already-validated backend workspace and
scope them to the requesting window/session. Recent history and archive preferences are not
capability stores.

Failure mechanism: data (`recent-dirs.json`/`archiveFolder`) and authority use the same mutable
contract. A compromised or buggy renderer promotes arbitrary data into a canonical root before
the otherwise-sound containment check.

Break-it angle: from renderer code call `addRecentDir(os-sensitive-existing-directory)`, then
invoke `readFile`, `writeFile`, or `deleteFile` on a descendant.

Impact: a renderer compromise can expand from the app data directory/recent user choices to
arbitrary local files. Electron isolation reduces direct Node access but this alternate IPC path
restores broad filesystem authority.

Recommended remediation: issue opaque, per-window directory grants only from the chooser or a
validated workspace; keep recent lists separate; validate archive destinations through the same
grant flow; revoke grants with window lifetime and workspace removal. Regression: calling
`addRecentDir` directly must never change file authorization.

Implementation assessment: **M**, `cross_process_coordination`, nominal `codex`; requires IPC,
window-lifecycle, chooser, workspace, and filesystem tests. Non-goal: replacing the existing
canonical path/symlink containment helper.

### AUD-GOS-003 — Workspace folder access is metadata-only and is not propagated into tool policy

Severity: **High**  
Confidence: **Confirmed**  
Evidence basis: source-evidenced  
Domain: State-Transition  
Status: new in this audit

Evidence:

- `crates/gosling-sdk-types/src/workspace.rs:109-127` gives every additional folder a typed
  `Read`/`ReadWrite` access value and gives output folders explicit paths.
- `crates/gosling/src/workspace/service.rs:308-327` returns folders only inside
  `WorkspaceSessionContext`; `PreparedWorkspaceSession` has no effective additional-directory or
  per-directory permission field (`service.rs:23-34`).
- `crates/gosling/src/acp/server/new_session.rs:195-218` persists the workspace snapshot but does
  not map workspace folders/output folders into `Session.additional_working_dirs` or a tool
  access policy.
- `crates/gosling/src/workspace/service.rs:330-334` implements read-only behavior as prompt text:
  `Reference read-only folders without modifying them.`
- `crates/gosling/src/permission/working_dir_scope_inspector.rs:45-58` is disabled unless the
  opt-in session flag is true and, when enabled, knows only `working_dir` plus
  `additional_working_dirs`. `Session::default` sets the flag false
  (`session_manager.rs:981-988`).

Observed: a folder marked `read` is not a backend authorization rule. Additional reference/source
and output roots are not propagated into the session's working-directory model. Tools/MCP roots
therefore cannot distinguish read from write, and the default session does not enforce a working
directory boundary at all.

Expected boundary: workspace creation produces an enforceable per-session path policy pinned with
the session: primary/output/read-write roots permit mutation as configured; read-only roots permit
reads but deny or explicitly gate mutation. Prompt guidance may explain the policy but cannot be
the policy.

Failure mechanism: the Workspace domain model terminates at the prompt/snapshot seam instead of
being translated into the existing session and permission structures.

Break-it angle: create a workspace with a valuable reference folder marked Read, ask the agent to
edit a file there, and observe that no access-specific backend rule exists to deny it. Conversely,
enable directory restriction and use an output folder outside the primary root; it is absent from
the inspector's allowed roots.

Impact: user-configured read-only reference data can be modified, and legitimate workspace output
or reference paths can be rejected or omitted by consumers that rely on session working roots.

Recommended remediation: add a canonical per-session folder-policy snapshot; map workspace roots
at new-session creation; teach local tool inspection and MCP root construction to distinguish read
from mutation; preserve the pinned policy on resume and deleted-workspace cases. Regression tests
must cover read allowed/write denied, output-folder write allowed, symlink/canonical traversal, and
workspace switching not mutating the active session.

Implementation assessment: **L**, `workflow_protocol`, nominal `multi-agent`; touches shared SDK
types, session migration, ACP creation/resume, permissions, MCP roots, and Desktop presentation.
Non-goal: making every external provider enforce a policy it cannot technically support; such
providers must fail visibly or require a constrained execution adapter.

### AUD-GOS-004 — Delegated roles inherit the parent's complete extension surface by default

Severity: **High**  
Confidence: **Confirmed**  
Evidence basis: source-evidenced  
Domain: Security  
Status: reconfirmed known-open finding ORCH-002 / AOC-019

Evidence:

- `crates/gosling/src/agents/platform_extensions/summon.rs:93-109` limits agent-file metadata to
  name, description, and model; no declared capability contract exists.
- The delegate tool explicitly documents `Omit to inherit all` at `summon.rs:454-470`.
- `summon.rs:1182-1206` starts with every extension enabled in the parent session and filters only
  when the model supplies an `extensions` argument.
- `summon.rs:193-215` discovers role files from repo-local `.gosling`, `.agents`, and `.claude`
  directories, while `summon.rs:1410-1431` always constructs the child as `GoslingMode::Auto` and
  `SessionType::SubAgent`.
- Fail-closed inspectors are denied rather than hung (`agent.rs:892-918`), which is a valid
  compensating control but does not reduce the initial extension surface.

Observed: prose declaring a reviewer/scout role is not an authority boundary. Unless the model
voluntarily sends an allowlist (including an empty one), the child receives the parent's complete
tool/extension surface and runs unattended in Auto mode.

Expected boundary: a role's permitted capabilities are explicit, deterministic, least-authority,
and enforced independently of model-authored delegate arguments or untrusted repo prose.

Failure mechanism: capability inheritance is the default transition, while narrowing is an
optional model decision. The source file that defines role intent cannot express an enforceable
allowlist.

Break-it angle: add a repo-local "reviewer" agent that asks to modify a file; delegate to it
without an `extensions` field and inspect the child's enabled extensions.

Impact: a compromised prompt or role file can turn an apparently advisory child into an
unattended actor with the parent's mutation/network/delegation capabilities. Parent scope and
fail-closed inspectors still limit some effects, so this is High rather than Critical.

Recommended remediation: add a versioned role capability policy, default built-in advisory roles
to no mutation extensions, intersect every requested tool with the parent and role allowlists,
and display the resolved child authority. Unknown/legacy role files should require an explicit
inherit-all choice rather than silently inheriting. Test role × adapter × mode matrices with fake
agents.

Implementation assessment: **L**, `governance_decision` + `workflow_protocol`, nominal
`human-owner` then `multi-agent`. Non-goal: allowing a child to gain an extension the parent does
not have.

### AUD-GOS-005 — Foreign transcript import silently drops malformed JSONL records

Severity: **Medium**  
Confidence: **Confirmed**  
Evidence basis: source-evidenced  
Domain: Input-Output-Path  
Status: reconfirmed IOP-GSL-001

Evidence: Claude Code uses `filter_map(...from_str(...).ok())` at
`crates/gosling/src/session/import_formats/claude_code.rs:18-27`; Codex repeats it at
`codex.rs:31-40`; Pi repeats it for all body records at `pi.rs:59-61`. Only a completely
unparseable Claude/Codex file or invalid Pi header fails.

Observed: a truncated or corrupt middle line disappears while the surviving records import as a
complete session. No skipped count or partial marker reaches the operator.

Expected boundary: strict import aborts on any malformed nonblank record, or an explicit lenient
mode reports exact dropped-line counts and marks the resulting session partial.

Failure mechanism: `filter_map(Result::ok)` conflates invalid data with absent/non-relevant data.

Break-it angle: insert one malformed line between two valid messages and import the transcript.

Impact: users resume an incomplete history and can make decisions without missing tool calls,
outputs, or instructions being visible.

Recommended remediation: share a line parser that records line number and parse result; default
to failure or require explicit partial-import consent; persist/import provenance and show it in the
session header. Regression: valid-corrupt-valid never produces an unqualified success.

Implementation assessment: **S**, `local_guardrail`, nominal `codex`. Non-goal: rejecting valid
format-specific event types that the converter intentionally ignores.

### AUD-GOS-006 — Imported sessions promote untrusted working directories and history to authority

Severity: **Medium**  
Confidence: **Confirmed**  
Evidence basis: source-evidenced  
Domain: Data-Integrity  
Status: reconfirmed IOP-GSL-002

Evidence:

- Claude Code harvests the first transcript `cwd` at
  `crates/gosling/src/session/import_formats/claude_code.rs:29-33`; Pi uses its header `cwd` at
  `pi.rs:33-37,194-200`.
- `crates/gosling/src/session/session_manager.rs:3873-3889` deserializes the converted session and
  passes `import.working_dir` directly to canonical session creation.
- The imported conversation is then persisted at `session_manager.rs:3915-3917`; there is no
  imported/untrusted provenance state in this transition.

Observed: transcript-controlled paths and model/user history become a normal resumable Gosling
session. The user's selection of the import file is the only gate.

Expected boundary: normalize and validate the path, show it before activation, require relinking
when unavailable/outside approved roots, and persist an imported-untrusted provenance marker that
prevents history from masquerading as locally-authored authority.

Failure mechanism: format conversion is treated as trust conversion. Syntactically valid foreign
data crosses directly into the canonical session state.

Break-it angle: import a session whose `cwd` points to a sensitive local directory and whose
history instructs the agent that prior authorization already exists; then resume.

Impact: workflow mis-scoping and prompt-injection leverage over later file/process actions. The
effect still requires import plus resume/tool execution, hence Medium.

Recommended remediation: add an import review step with a user-selected effective directory,
canonical path validation, and durable provenance; do not activate tools until the location is
accepted. Regression: transcript `cwd` never becomes active without validation/confirmation.

Implementation assessment: **M**, `workflow_protocol`, nominal `codex` with UX review.
Non-goal: stripping foreign conversation content that the user intentionally wants to inspect.

### AUD-GOS-007 — The settings IPC erases every value contract except language

Severity: **Medium**  
Confidence: **Confirmed**  
Evidence basis: source-evidenced  
Domain: Architecture  
Status: reconfirmed ARCN-GSL-003

Evidence: `ui/desktop/src/preload.ts:287-293` advertises `Settings[K]`, but the main handler receives
`value: unknown` at `main.ts:2036`; after validating only key membership and language, lines
2048-2051 assign `(settings as any)[key] = value` and persist it. The settings type contains nested
objects, arrays, nullable strings, booleans, and constrained enums (`utils/settings.ts:49-71`).

Observed: a buggy or compromised renderer can persist `theme: 42`,
`keyboardShortcuts: "oops"`, malformed external-backend configuration, or attacker-shaped
secret-profile data under a valid key.

Expected boundary: runtime per-key schemas at the privileged IPC seam, including nested size and
shape limits, match the compile-time contract and reject invalid values without changing disk.

Failure mechanism: TypeScript generics exist only in renderer compilation; the main process
re-establishes neither type nor semantic invariants.

Break-it angle: invoke `set-setting` with each valid key and a value of the wrong primitive/nested
shape, then restart or trigger its immediate consumer.

Impact: durable settings corruption, startup/runtime failures, and attacker-controlled data
reaching security-sensitive consumers such as filesystem roots and backend trust configuration.

Recommended remediation: use a discriminated runtime validator map in main; cap collections and
strings; return typed validation errors; test every key with valid, wrong-type, oversized, and
prototype-shaped values. Implementation: **S-M**, `local_guardrail`, nominal `codex`.

### AUD-GOS-008 — Desktop settings writes can tear and silently reset the whole profile

Severity: **Medium**  
Confidence: **Confirmed** for the non-atomic mechanism; runtime interruption not observed  
Evidence basis: source-evidenced  
Domain: Reliability  
Status: new in this audit

Evidence:

- `ui/desktop/src/main.ts:375-382` and `2036-2051` overwrite `settings.json` directly with
  `writeFileSync`; the migration path does the same at lines 225-230. There is no unique temp,
  `fsync`, atomic rename, directory sync, or restrictive mode.
- `main.ts:214-234` catches any parse/read failure and returns all defaults. It does not quarantine,
  recover, or surface a user-visible corruption state.
- `ui/desktop/src/utils/recentDirs.ts:40-50,74-84` repeats direct whole-file replacement and empty
  fallback for the recent-directory authority list.

Observed: settings and recent-directory records use truncate-and-rewrite persistence. A process,
power, disk-full, or write error can leave partial JSON; the next launch silently treats it as an
empty/default profile.

Expected boundary: owner-only atomic write (unique temp in the same directory, flush, rename,
directory sync), previous-good recovery/quarantine, and a visible recovery message. The main
settings writer should reuse Gosling's established atomic persistence pattern.

Failure mechanism: the durable record is destroyed before its replacement is known durable, and
parse failure is classified as absence.

Break-it angle: terminate the main process or force an I/O error during the write, then relaunch
and inspect whether the prior profile is recovered or silently defaulted.

Impact: loss of Desktop configuration, archive/session preferences, backend connection/trust
configuration, and recent roots. Values can be re-entered, but the reset is workflow-wide and
silent.

Recovery/FMECA: normal-run file replacement → partial JSON → next-start default fallback;
detection is delayed until user observation or a console log; current safe state is neither
fail-visible nor rollback. Target safe state: `fail_rollback` with quarantine and notification.

Recommended remediation: one owner-only atomic JSON helper used by settings and recent dirs,
schema validation before rename, `.previous`/quarantine recovery, and interruption/fault-injection
tests. Implementation: **S-M**, `persistence_recovery`, nominal `codex`.

### AUD-GOS-009 — Session import materializes unbounded local and remote payloads

Severity: **Low**  
Confidence: **Confirmed** for the missing bound; exhaustion not reproduced  
Evidence basis: source-evidenced  
Domain: Input-Output-Path  
Status: reconfirmed IOP-GSL-004

Evidence: CLI import uses unbounded `fs::read_to_string` at
`crates/gosling-cli/src/commands/session.rs:297-308`; Desktop returns an unbounded
`fs.readFile(filePath, 'utf8')` over IPC at `ui/desktop/src/main.ts:2330-2349`; Nostr import
decrypts the complete remote event content into a `String` at
`crates/gosling/src/session/nostr_share.rs:226-236`.

Observed: there is no byte cap before whole-payload allocation and parsing.

Expected boundary: stat/stream with a documented maximum before materialization; remote content is
bounded before/while fetch/decrypt; the same cap and error shape apply to CLI and Desktop.

Failure mechanism: caller/relay-controlled size drives main-process or CLI heap allocation.

Break-it angle: select a multi-gigabyte JSONL file or serve an oversized share event.

Impact: local responsiveness loss or OOM. Recommended repair: shared import-byte budget plus
oversize tests. Implementation: **S**, `local_guardrail`, nominal `codex`.

### AUD-GOS-010 — Invalid server host configuration panics during startup

Severity: **Low**  
Confidence: **Confirmed**  
Evidence basis: source-evidenced  
Domain: Reliability  
Status: new in this audit

Evidence: `crates/gosling-server/src/configuration.rs:19-22` returns `SocketAddr` by parsing
`host:port` and calling `expect("Failed to parse socket address")`. `Settings::new` otherwise
returns typed configuration errors (`configuration.rs:25-70`), but `commands/agent.rs:128` calls
the panicking method during startup.

Observed: an invalid `GOSLING_HOST` (including a hostname where a numeric `SocketAddr` is required)
terminates via panic instead of a configuration error.

Expected boundary: validate host/bind configuration during `Settings::new`, return a typed error
that names the setting, and exit normally with an actionable message.

Failure mechanism: a fallible operator-input parse is encoded as an invariant with `expect`.

Break-it angle: start with `GOSLING_HOST=not-an-address` or `localhost` and inspect the panic path.

Impact: local/service startup outage with a backtrace-style failure instead of guided recovery.
Recommended repair: make `socket_addr` return `Result`, or model host as `IpAddr`; test invalid and
hostname cases. Implementation: **XS**, `local_guardrail`, nominal `codex`.

## What would falsify the strongest conclusion?

AUD-GOS-001 would be falsified if the apparent `ManagedSecretEntry.value` were a non-secret opaque
reference and a backend boundary replaced it with a protected value only at a non-model operation.
The current source proves the opposite: UI copy calls the entries passwords/admin credentials,
the raw value is rendered, persisted, concatenated as `KEY=value`, annotated specifically for the
assistant, stored as a normal message, traced, handed to hooks, and retained by the provider-facing
audience filter. A runtime trace is unnecessary to establish that deterministic dataflow; a
runtime sentinel test would measure which optional sinks are enabled in a particular deployment.
