# gosling v1.0.0

gosling v1.0.0 establishes the first stable release line for the independently maintained gosling project. It combines the Rust agent core, Electron Desktop, CLI and text interfaces, ACP/provider adapters, MCP extensions, local session persistence, workspace-aware chats, and hardened execution boundaries.

This page prepares the release content; it does not prove that the `v1.0.0` tag or artifacts have been published. The [GitHub release](https://github.com/repo-makeover/gosling/releases/tag/v1.0.0), downloaded artifact metadata, and runtime version output are authoritative after publication.

## Highlights

- Workspace rows filter existing chats without changing the global New Chat default.
- Starting a chat from a workspace action preselects that workspace; global New Chat keeps workspace selection explicit.
- The chat composer includes a credential-profile selector and manager, and existing sessions retain their pinned workspace and credential identity.
- Desktop startup, shutdown, stale-backend cleanup, packaged loopback connectivity, single-instance behavior, and native multi-window actions have focused regression evidence.
- Session interruption, provider failure exits, machine-readable CLI output, malformed configuration, doctor behavior, empty prompts, plugin validation, and ACP lifecycle handling were repaired through the all-scenario playtest campaign.
- Local summarization, durable facts, compacted-session resume paging, and bounded context-management paths support longer sessions.
- Tool inspection, secret storage, path handling, plugin installation, provider clients, and release workflows use tighter fail-closed or least-privilege behavior.

## Desktop workspaces and credentials

Workspaces are repeatable Desktop environments with a primary working folder, optional source/reference folders, typed output destinations, provider/model defaults, and secure credential-profile bindings.

Selecting a workspace in the sidebar filters its chats. It does not mutate an existing session or make every later New Chat use that workspace. The workspace `+` action is the intentional shortcut that opens New Chat with that workspace preselected.

Credential values remain in the OS keyring or owner-protected fallback rather than workspace metadata or renderer responses. The chat composer can display/select profiles and open the manager; missing profiles fail visibly and require relinking instead of silently falling back to another account.

See the [workspace manual](/docs/guides/workspaces) for session pinning, output routing, import/export boundaries, persistence, and current limits.

## Reliability and validation evidence

The [2026-07-20 live playtest](https://github.com/repo-makeover/gosling/blob/main/docs/cloud/2026-07-20-live-all-scenarios-playtest.md) executed all 110 scenario cards. Its original ledger records 46 pass, 32 fail, and 32 blocked. The appended repair closure records repairs and focused verification for all 14 original findings plus a fifteenth installed-Desktop windowing defect found during replay.

That closure is not represented as a full 110-card post-repair rerun. The release owner must complete the repository [v1.0.0 checklist](https://github.com/repo-makeover/gosling/blob/main/RELEASE_CHECKLIST.md), including full Rust, Desktop, docs, Clippy, packaged-GUI, signing, checksum, clean-install, and scenario gates before publication.

## Security and privacy

- Safety-inspector errors require approval instead of silently allowing a tool call.
- Session, token, secret, and workspace persistence use owner-restricted paths and atomic writes where applicable.
- Cache and memory paths reject traversal outside their intended roots.
- Plugin clone inputs are protected from command-option injection.
- Provider HTTP clients and retry behavior are bounded.
- Workspace metadata stores profile references and non-secret status, not credential values.
- Retired write-capable workflows are not described as part of the current patch-release path.

Security behavior still depends on the selected permission mode, provider, enabled extensions, and user-approved tool scope. Review permissions before running gosling against untrusted repositories or third-party MCP servers.

## Compatibility and provenance

gosling descends from goose v1.38 and preserves inherited Apache 2.0 authorship and commit history. It is independently maintained and is not endorsed by or affiliated with goose, AAIF, or the Linux Foundation.

gosling and goose use separate application identities, configuration/data locations, session databases, keyring services, deep-link schemes, and single-instance locks. Goose catalogs may still appear through documented deterministic compatibility adapters that preserve source provenance.

## Upgrade notes and current limits

- Confirm the installed version with `gosling --version` and Desktop **Help > About**.
- Local model runtimes are not bundled; configure a supported external provider or local service.
- Workspaces and workspace credential profiles are Desktop-only in v1.0.0.
- `tagteam` and subscription-backed vendor CLIs must be installed and authenticated separately.
- Official Homebrew formula/cask availability is not claimed by these notes.
- Keep a recoverable copy of important configuration and session data before any major-version upgrade.

## Release invariant

The final tag must not be created until the Rust workspace, Cargo lockfile, Desktop package and lockfile, OpenAPI/generated metadata, packaged app, About surface, and runtime version output all report `1.0.0`. Signing, publication, updater promotion, and announcement remain maintainer-owned actions.
